전체 코드를 다 읽었습니다. 상세하게 분석하겠습니다.

전체 구조 평가
해커톤 프로젝트치고 아키텍처가 상당히 잘 잡혀 있습니다. 프론트엔드는 React + wouter + Tailwind + Framer Motion, 백엔드는 Express + SSE 기반 실시간 통신, 상태 관리는 Context API 두 개(Timer, Social)로 분리되어 있고, 로컬 통계와 서버 통계를 이중으로 추적하는 구조입니다. 기능 범위 대비 파일 수가 적고 모듈 경계가 명확합니다.
실제로 문제가 되는 부분들:
1. 백엔드 데이터가 전부 인메모리입니다.
teamStore.ts와 statsStore.ts 모두 Map에 데이터를 저장하고 있어서, 서버 재시작 시 모든 팀, 멤버, 통계가 소멸합니다. Replit은 일정 시간 비활성 시 컨테이너를 내리기 때문에 실사용에서 팀 코드 공유 후 다음 날 접속하면 팀이 사라져 있는 상황이 발생합니다. 해커톤 시연용이라면 상관없지만, 만약 계속 쓸 거라면 SQLite나 최소한 JSON 파일 persist가 필요합니다.
2. TimerContext의 초기 phase가 'focusing'으로 하드코딩되어 있습니다.

const [phase, setPhase] = useState<TimerPhase>('focusing');


앱을 처음 열면 사용자가 아무것도 누르지 않았는데 “집중 중…” 상태로 시작하고, 타이머가 즉시 카운트다운을 시작합니다. 이건 'idle'이어야 합니다. 현재는 앱을 열자마자 의도치 않게 세션이 돌아가고, workDuration 후에 NOW! 알림이 터집니다. start() 함수에서 'focusing'으로 전환하도록 되어 있으니, 초기값만 'idle'로 바꾸면 됩니다. 단, endTimeMs와 remainingSeconds의 초기값도 함께 조정해야 카운트다운 interval이 즉시 실행되지 않습니다.
3. SocialContext의 connectTeam 함수가 클로저 참조 문제를 가지고 있습니다.
connectTeam은 컴포넌트 내부의 일반 함수인데, useEffect 안에서 memberships가 바뀔 때마다 호출됩니다. 문제는 이 함수가 setAllMembers, setPeerAlertMsg, setPokeFrom 같은 setter를 직접 캡처하고 있으면서, useCallback이나 useRef로 안정화되지 않았다는 점입니다. React Strict Mode에서 이중 마운트될 경우, 같은 팀 코드에 대해 EventSource가 두 번 연결될 수 있습니다. esMap.current.has(m.code) 체크가 있긴 하지만, cleanup 타이밍에 따라 race condition이 발생할 수 있습니다.
4. useStatsTracker에서 dependency 없는 useEffect를 사용하고 있습니다.

useEffect(() => {
  // ... phase와 ignoreLevel 변화 추적 ...
});


dependency array가 아예 없어서 매 렌더마다 실행됩니다. 의도적으로 “모든 렌더에서 이전 값과 비교”하는 패턴을 쓴 건 이해하지만, 이게 StatsPage에서 recharts가 리렌더를 자주 일으키거나 다른 context 변화가 있을 때 불필요하게 계속 실행됩니다. [phase, ignoreLevel]을 dependency로 주고 내부에서 ref로 이전 값을 추적하는 방식이 더 안전합니다.
5. sounds.ts의 Lv.3 루프가 onended 기반인데, 이게 모바일 Safari에서 불안정합니다.

a.onended = () => {
  if (isLooping) {
    a.currentTime = 0;
    a.play().catch(() => {});
  }
};


iOS Safari에서는 onended 후 play()가 사용자 인터랙션 없이 재생되지 않는 경우가 있습니다. HTMLAudioElement.loop = true 속성을 쓰는 게 모바일에서 더 안정적입니다. Lv.3에서 a.loop = true; a.play()로 바꾸고, stopAlert에서 a.loop = false로 리셋하면 됩니다.
6. 설정 페이지의 “저장” 버튼이 실제로 아무것도 하지 않습니다.

const handleSave = () => {
  setSaved(true);
  setTimeout(() => setSaved(false), 2000);
};


updateSettings는 이미 각 NumberInput의 onChange에서 호출되어 localStorage에 즉시 저장되고 있습니다. “저장” 버튼은 시각적 피드백만 주고 실제 저장 동작이 없습니다. 사용자 입장에서는 “저장을 눌러야 적용되나?“라는 혼란을 줄 수 있습니다. 버튼을 제거하고 “자동 저장” 표시를 넣거나, 아니면 실제로 pending state를 만들어서 버튼을 눌러야 저장되게 하거나 둘 중 하나로 통일해야 합니다.
7. 팀 멤버 인증이 약합니다.
memberToken은 randomBytes(16).toString('hex')로 생성되어 충분히 강력하지만, 팀 코드 자체가 6자리 영숫자(ABCDEFGHJKLMNPQRSTUVWXYZ23456789, 31^6 ≈ 8.9억 조합)입니다. 해커톤용으로는 충분하지만, 팀 코드를 알면 누구나 join할 수 있고, GET /api/teams/:code는 인증 없이 팀 전체 멤버 정보(닉네임, 상태, 통계)를 반환합니다. 실서비스로 갈 경우 이 엔드포인트에 최소한 rate limiting이 필요합니다.
8. todayKst() 함수가 프론트/백에서 다른 포맷을 씁니다.
프론트의 statsStorage.ts는 en-CA 로케일로 YYYY-MM-DD 포맷을, 백엔드의 teamStore.ts는 ko-KR 로케일로 YYYY. MM. DD. 포맷을 씁니다. 백엔드 statsStore.ts는 다시 en-CA를 씁니다. teamStore에서 날짜 비교(member.todayDate !== today)가 실행될 때, 이 불일치가 nowCount/dismissedCount 일일 리셋 로직에 영향을 줄 수 있습니다. teamStore의 todayKst를 en-CA 포맷으로 통일해야 합니다.
9. package.json에 사용하지 않는 의존성이 대량으로 있습니다.
Radix UI 컴포넌트가 20개 가까이 devDependencies에 있는데, 실제 import하는 곳이 없습니다. shadcn/ui 보일러플레이트에서 그대로 가져온 것으로 보입니다. 번들 사이즈에는 tree-shaking으로 영향 없겠지만, node_modules 설치 시간과 lock file 복잡도에 영향을 줍니다.
잘 된 부분:
에스컬레이션 시스템 설계가 핵심 기능에 대한 기획 의도가 명확하게 코드에 반영되어 있습니다. Lv.1→2→3으로 올라가면서 볼륨, 반복 횟수, 시각적 강도가 동시에 바뀌고, Lv.3에서 스누즈를 제거하는 판단이 UX적으로 맞습니다. SSE 기반 실시간 동기화도 WebSocket 대신 SSE를 선택한 게 이 규모에서는 적절하고, heartbeat + 자동 재연결까지 구현되어 있습니다. 통계 추적의 이중 구조(로컬 + 서버)도 오프라인 퍼스트 접근으로서 올바른 방향입니다.​​​​​​​​​​​​​​​​
